You are working in a Next.js + TypeScript + Drizzle codebase.

  Goal:
  Build the actual execution engine agent system layer, not the UI shell. The current /ops page is only the control
  room. The real work is to create a durable agent runtime underneath it.

  Core principle:
  No in-memory state as source of truth. All campaign, agent, event, artifact, blocker, attempt, and resume state must
  be durable in Postgres.

  What already exists:
  - A /ops dashboard and related API routes
  - A mission-control schema with tables like agents, campaigns, campaign_agents, campaign_events, blockers,
  execution_attempts, resume_directives, artifacts
  - A workflow layer that currently has placeholder step functions and a thin runtime bridge

  What to build:
  1. Durable agent model
  - Create a first-class agent entity model persisted in Postgres.
  - Agents need identity, display name, role, capabilities, status, ownership, and metadata.
  - Support lead agents and worker agents.
  - Support agent lifecycle states like active, blocked, unavailable, retired, and running.

  2. Durable execution engine
  - Build the agent execution system that actually runs work.
  - It should handle:
    - campaign creation
    - agent assignment
    - tool execution
    - event emission
    - artifact writes
    - blocker creation and resolution
    - resume directives
    - execution attempts
    - status transitions
  - Persist everything durably.
  - Support restart and rehydration from the database after server reload.

  3. Real tool layer
  - Add a tool registry and adapters for:
    - file writes
    - artifact creation
    - web/browser extraction
    - MCP tool calls
    - database reads/writes when appropriate
  - If a tool is not actually available, return a deterministic structured failure, not fake success.
  - The agent loop should be able to recover, retry, or escalate.

  4. Agent memory and traces
  - Persist per-campaign and per-agent memory, not just a single feed.
  - Keep a full audit trail of:
    - prompts
    - tool calls
    - tool results
    - status changes
    - blockers
    - resumes
    - retries
  - The feed should be a projection of this durable history.

  5. Orchestration and handoff
  - Support a minimal multi-agent handoff model.
  - One lead agent may dispatch work to worker agents.
  - The orchestration layer should know how to:
    - start a campaign
    - assign work
    - pause on blockers
    - resume after approval
    - restart failed attempts
    - mark completion only when required outputs exist

  6. Service layer first
  - Put the logic in a dedicated service layer, not inside API routes or React components.
  - Keep the UI contract stable.
  - The service should own reads, writes, normalization, and durable event appends.
  - The page and API routes should only consume the service.

  7. Demo seed behavior
  - Demo campaigns must be durable and marked as demo in metadata.
  - Demo deletion must remove only demo-marked records.
  - Do not rely on name prefixes as the only marker.

  8. Keep the existing API surface stable for now
  - Preserve the current /ops routes unless a route is clearly broken.
  - Keep response shapes stable where possible.
  - If you need to add new fields, add them compatibly.

  Non-goals for this task:
  - Do not redesign the /ops UI.
  - Do not spend time on visual polish.
  - Do not build a fake in-memory simulator.
  - Do not build only scaffolding or type definitions.
  - Do not stop at "the code compiles" if the system cannot survive reloads.

  Implementation guidance:
  - Start by auditing the current ops and mission-control files.
  - Create a clean durable service layer for agents and campaigns.
  - Wire the workflow step functions to the durable service.
  - Wire the runtime bridge to the durable service.
  - Replace any in-memory registry logic with Postgres-backed reads and writes.
  - Make sure campaign feeds, artifacts, blockers, and execution attempts survive reloads.
  - Ensure the system can be restarted and still rehydrate the same state from Postgres.

  - Create campaign
  - Assign lead agent
  - Start execution attempt
  - Append feed event
  - Write artifact
  - Create blocker
  - Resolve blocker
  - Mark completed
  - Reload and confirm state is still there
